关键概念

离线 vs 实时 · 连续帧 · 时间约束

范式转变 · Paradigm Shift

从离线图片检测到实时检测

图片检测处理单个静态输入，程序运行一次即结束。视频检测处理连续帧流，程序必须持续运行并满足时间约束。

图片检测（离线）

输入：单张图片
输出：检测结果
耗时：无硬性约束
资源：用完即释放
典型场景：批量审核、相册分类

视频检测（实时）

输入：连续帧流
输出：逐帧结果 + 时序稳定性
耗时：必须 ≤ 帧间隔
资源：持续占用摄像头/GPU
典型场景：安防监控、自动驾驶

核心问题：如果模型单帧推理耗时 100ms，理论最大 FPS 为 10。但实际还需扣除读取、预处理、绘制和显示时间，真实 FPS 通常更低。

关键概念

帧率 · 时间维度 · 实时性约束

基础概念 · Fundamentals

视频的本质：连续帧与时间维度

视频可视为每秒若干张静态图片的序列。帧率（FPS, Frames Per Second）描述每秒显示的帧数。

常见帧率标准

24 FPS — 电影工业标准
25 FPS — PAL 制式（中国电视）
30 FPS — NTSC 制式
60 FPS — 高帧率显示
120+ FPS — 高速摄像

实时检测的时间约束

25 FPS → 每帧可用时间 40ms
30 FPS → 每帧可用时间 33ms
60 FPS → 每帧可用时间 16.7ms
推理耗时必须 ≤ 帧间隔，否则产生卡顿

关键洞察：实时检测系统不仅要考虑推理速度，还需考虑读取、预处理、后处理、绘制、显示等全部环节的累积耗时。管线中任一环节成为瓶颈，整体 FPS 都会受限。

关键 API

VideoCapture · read() · waitKey()

OpenCV · Video I/O

OpenCV 摄像头读取基本流程

OpenCV 通过 VideoCapture 类提供统一的视频/摄像头读取接口。

import cv2

# 初始化摄像头（0=默认摄像头，也可传入视频文件路径）
cap = cv2.VideoCapture(0)

while True:
    # 读取一帧：返回 (是否成功, 帧数据)
    ok, frame = cap.read()
    if not ok:
        break  # 读取失败则退出循环

    # 显示帧
    cv2.imshow("camera", frame)

    # waitKey(1) 等待 1ms 键盘输入，同时驱动窗口刷新
    if cv2.waitKey(1) & 0xFF == ord("q"):
        break

# 释放资源（必须执行，否则摄像头可能被占用）
cap.release()
cv2.destroyAllWindows()

CAP_PROP_* · 分辨率 · 缓存

OpenCV · 参数配置

VideoCapture 关键参数与常见问题

# 设置摄像头参数（需在 read() 之前设置）
cap = cv2.VideoCapture(0)
cap.set(cv2.CAP_PROP_FRAME_WIDTH, 1280)
cap.set(cv2.CAP_PROP_FRAME_HEIGHT, 720)
cap.set(cv2.CAP_PROP_FPS, 30)
cap.set(cv2.CAP_PROP_BUFFERSIZE, 1)  # 减少缓存延迟

# 读取实际参数（摄像头可能不支持设定值）
w = cap.get(cv2.CAP_PROP_FRAME_WIDTH)
h = cap.get(cv2.CAP_PROP_FRAME_HEIGHT)
fps = cap.get(cv2.CAP_PROP_FPS)
print(f"实际分辨率: {w}×{h}, FPS: {fps}")
常见问题原因解决方案
摄像头无法打开设备号错误 / 被其他进程占用尝试 cap = VideoCapture(0, cv2.CAP_DSHOW)（Windows）
读取延迟高内部缓存队列堆积设置 CAP_PROP_BUFFERSIZE=1
分辨率不匹配摄像头不支持设定值读取 get() 实际值并适配
cap.release() 后仍被占用进程未完全退出检查是否有 Python 进程残留

帧采集：摄像头与视频文件

采集阶段从摄像头或视频文件读取原始帧。采集速度受硬件接口、分辨率和驱动影响。

摄像头采集

分辨率建议：1280×720 或 640×480

缓存：默认队列可能堆积旧帧

视频文件读取

速度：通常快于摄像头（无传输延迟）

注意：解码耗时不可忽略

预处理

Letterbox · 归一化 · 数据拷贝

管线阶段 02 · 预处理

Letterbox 与归一化

预处理将原始帧转换为模型所需的输入格式，包括尺寸调整、归一化和数据类型转换。

def preprocess(frame, input_size=640):
    # Letterbox：保持宽高比的缩放 + 填充
    h, w = frame.shape[:2]
    scale = input_size / max(h, w)
    nw, nh = int(w * scale), int(h * scale)
    resized = cv2.resize(frame, (nw, nh))

    # 创建画布并居中放置
    canvas = np.full((input_size, input_size, 3), 114, dtype=np.uint8)
    dx = (input_size - nw) // 2
    dy = (input_size - nh) // 2
    canvas[dy:dy+nh, dx:dx+nw] = resized

    # 归一化 + HWC→CHW + 添加 batch 维度
    blob = canvas.astype(np.float32) / 255.0
    blob = blob.transpose(2, 0, 1)[np.newaxis, ...]
    return blob, (scale, dx, dy)  # 返回缩放参数供坐标还原
性能注意：频繁的 numpy 数组拷贝和 cv2.resize 是预处理阶段的主要耗时来源。使用 np.ascontiguousarray() 可减少内存重排开销。

管线阶段 03 · 模型推理

前向传播与推理引擎

模型推理是管线中最耗时的阶段。推理引擎的选择直接影响速度和部署便利性。

PyTorch 原生推理

优点：开发便捷，调试友好

适用：原型验证、研究阶段

后处理

NMS · 阈值过滤 · 候选框

管线阶段 04 · 后处理

NMS 与阈值过滤

模型输出大量候选框，需通过置信度阈值和 NMS（非极大值抑制）筛选出最终检测结果。

def postprocess(outputs, conf_thres=0.25, iou_thres=0.45):
    # 解析输出：假设 outputs shape 为 (1, 8400, 6)
    pred = outputs[0]  # (8400, 6)

    # 步骤1：置信度过滤
    mask = pred[:, 4] > conf_thres
    pred = pred[mask]

    if len(pred) == 0:
        return np.empty((0, 6))

    # 步骤2：NMS — 按置信度排序，消除重叠框
    boxes = pred[:, :4]  # [x1, y1, x2, y2]
    scores = pred[:, 4]
    indices = cv2.dnn.NMSBoxes(boxes.tolist(), scores.tolist(),
                               conf_thres, iou_thres)

    if len(indices) == 0:
        return np.empty((0, 6))
    indices = indices.flatten()
    return pred[indices]
NMS 性能：候选框数量直接影响 NMS 耗时。设置合理的置信度阈值（如 0.25）可大幅减少候选框数量。对于 8400 个候选框，NMS 耗时通常在 1-5ms。

管线阶段 05 · 坐标还原与绘制

将检测框映射回原图

模型输出的坐标基于预处理后的输入尺寸，需通过 letterbox 的缩放参数还原到原始帧坐标。

def scale_boxes(boxes, scale, dx, dy, orig_shape):
    # 将 letterbox 坐标还原到原图坐标
    boxes = boxes.copy()
    boxes[:, [0, 2]] -= dx  # 减去 padding
    boxes[:, [1, 3]] -= dy
    boxes /= scale  # 除以缩放比
    # 裁剪到图像边界
    h, w = orig_shape[:2]
    boxes[:, [0, 2]] = boxes[:, [0, 2]].clip(0, w)
    boxes[:, [1, 3]] = boxes[:, [1, 3]].clip(0, h)
    return boxes

def draw_detections(frame, dets, class_names):

性能指标 · Metrics

FPS 计算与测量方法

FPS（每秒处理帧数）是衡量实时检测系统流畅度的核心指标。

# 滑动平均 FPS 测量
fps_values = []
prev_time = time.perf_counter()

while True:
    ok, frame = cap.read()
    if not ok: break

    # 推理管线...
    dets = run_inference(frame)

    # FPS 计算
    now = time.perf_counter()
    fps = 1.0 / (now - prev_time)
    prev_time = now

    # 滑动平均（平滑波动）
    fps_values.append(fps)
    if len(fps_values) > 30:
        fps_values.pop(0)
    avg_fps = sum(fps_values) / len(fps_values)

    # 显示
    cv2.putText(frame, f"FPS: {avg_fps:.1f}", ...)
测量注意事项：FPS 测量应包含整个管线耗时（从读取到显示），而非仅推理时间。滑动平均窗口建议 30 帧，可有效平滑瞬时波动。

性能指标 · Latency & Throughput

延迟与吞吐量

延迟（Latency）描述单帧从输入到输出的耗时，吞吐量（Throughput）描述系统单位时间的处理能力。

指标定义公式适用场景
FPS每秒处理帧数processed / elapsed实时显示流畅度
Latency单帧端到端耗时t_end - t_start响应速度敏感场景
Throughput单位时间处理量batch / time批量离线检测
Pipeline 延迟各阶段耗时分解Σ stage_time瓶颈定位
重要区分：高 FPS 不一定意味着低延迟。异步管线可能通过队列缓冲提高 FPS，但单帧延迟反而增加。对于实时交互场景，延迟比 FPS 更关键。

跳帧 · 帧间复用

优化策略 03 · 跳帧检测

跳帧检测策略

对于目标移动较慢的场景（如安防监控），无需每帧都执行完整推理。

def skip_frame_detection(cap, skip=2):
    frame_count = 0
    last_dets = []

    while True:
        ok, frame = cap.read()
        if not ok: break

        if frame_count % skip == 0:
            # 执行完整推理
            last_dets = run_inference(frame)
        # 跳过的帧复用上次检测结果
        draw_detections(frame, last_dets)

        frame_count += 1
        cv2.imshow("skip", frame)
        if cv2.waitKey(1) & 0xFF == ord("q"):
            break
效果：skip=2 时推理量减半，FPS 可提升约 80%。skip=3 时推理量降为 1/3，FPS 提升约 150%。适用于目标运动速度 ≤ 检测区域宽度 5%/帧 的场景。

ROI · 区域裁剪

优化策略 04 · 限定 ROI

限定检测区域（ROI）

如果目标只可能出现在画面特定区域（如门口、传送带），可只对该区域进行检测。

ROI 裁剪实现

1. 定义 ROI 区域坐标
2. 从原帧裁剪 ROI 子图
3. 对子图执行预处理 + 推理
4. 将检测框坐标映射回原图
计算量降低比例 ≈ ROI面积/全图面积

适用场景

• 闸机口人脸检测
• 生产线缺陷检测
• 停车场车位检测
• 交通灯区域检测
注意：ROI 外目标完全丢失

# ROI 检测示例
roi_x1, roi_y1, roi_x2, roi_y2 = 200, 100, 600, 500
roi = frame[roi_y1:roi_y2, roi_x1:roi_x2]  # 裁剪 ROI
dets = run_inference(roi)  # 在 ROI 上检测
# 坐标还原：dets.x += roi_x1, dets.y += roi_y1

稳定性

抖动 · 波动 · 漏检

检测稳定性 · Stability

实时检测中的常见稳定性问题

实时画面中，检测结果在时间维度上可能出现不稳定现象，影响系统可靠性。

检测框抖动

相邻帧之间检测框位置和大小剧烈变化。原因：模型对微小位移敏感、输入噪声。

置信度波动

同一目标在不同帧的置信度忽高忽低，可能导致目标在阈值上下反复出现/消失。

目标偶尔漏检

目标在部分帧中完全未被检测到。原因：遮挡、运动模糊、光照变化。

相邻目标 ID 混乱

多目标场景下，同一目标在不同帧被分配不同 ID。需要跟踪算法解决。

import cv2, time, numpy as np
import onnxruntime as ort

# 加载模型
session = ort.InferenceSession("best.onnx")
input_name = session.get_inputs()[0].name

cap = cv2.VideoCapture(0)
cap.set(cv2.CAP_PROP_BUFFERSIZE, 1)
fps_log, prev = [], time.perf_counter()

while True:
    ok, frame = cap.read()
    if not ok: break

    # 预处理 → 推理 → 后处理
    blob, meta = preprocess(frame, 640)
    outputs = session.run(None, {input_name: blob})
    dets = postprocess(outputs)

    if len(dets):
        dets[:, :4] = scale_boxes(dets[:, :4], *meta, frame.shape)
        draw_detections(frame, dets)

    # FPS
    now = time.perf_counter()
    fps = 1.0 / (now - prev); prev = now
    fps_log.append(fps)
    if len(fps_log) > 30: fps_log.pop(0)
    cv2.putText(frame, f"FPS: {sum(fps_log)/len(fps_log):.1f}",
                (10, 30), cv2.FONT_HERSHEY_SIMPLEX, 1, (0,255,0), 2)

    cv2.imshow("detect", frame)
    if cv2.waitKey(1) & 0xFF == ord("q"): break

cap.release(); cv2.destroyAllWindows()

from fastapi import FastAPI, UploadFile, File
import cv2, numpy as np

app = FastAPI()
session = ort.InferenceSession("best.onnx")

@app.post("/detect")
async def detect(file: UploadFile = File(...)):
    # 读取上传图片
    contents = await file.read()
    nparr = np.frombuffer(contents, np.uint8)
    img = cv2.imdecode(nparr, cv2.IMREAD_COLOR)

    # 推理
    blob, meta = preprocess(img, 640)
    outputs = session.run(None, {input_name: blob})
    dets = postprocess(outputs)

    # 绘制结果
    if len(dets):
        dets[:, :4] = scale_boxes(dets[:, :4], *meta, img.shape)
        draw_detections(img, dets)

    # 返回结构化结果
    return {
        "detections": dets.tolist(),
        "num_objects": len(dets)
    }
