Context · 项目上下文

课程项目：目标检测平台

本课程项目开发一个基于 YOLO 模型的目标检测平台，支持图片与视频上传、模型推理、结果展示与记录管理。技术栈如下：

  • 前端：HTML / CSS / JavaScript 或 React，提供直观的文件上传与结果展示界面
  • 后端：Python FastAPI，处理文件接收、模型调用、结果返回
  • 模型推理：YOLOv5 / YOLOv5-lite，加载预训练权重执行目标检测
  • 存储：本地文件系统存储上传文件与检测结果，SQLite 记录检测日志
用户浏览器 ──HTTP──→ FastAPI 后端 ──调用──→ YOLO 模型推理 ──→ 结果返回浏览器 │ │ └── 上传 jpg/png/mp4 ┘ └── 保存至 uploads/ + 记录至 SQLite
项目规模参考

前端约 3-5 个页面（上传页、结果页、历史页），后端约 4-6 个 API 端点，模型推理封装为独立模块。

开发环境

Python 3.10+ · FastAPI 0.104+ · PyTorch 2.0+ · YOLOv5-lite · 推荐使用 conda 虚拟环境隔离依赖。

AI Tool Positioning

AI 工具定位与能力边界

AI 适合承担的任务
  • 需求拆解：将模糊描述转化为结构化功能清单
  • 代码骨架：生成接口模板、数据模型、目录结构
  • 编译错误解释：分析错误堆栈并给出修复建议
  • 接口调用示例：生成 curl 命令、客户端调用代码
  • 文档初稿：API 文档、README、使用说明
  • 测试用例生成：基于函数签名生成边界测试
AI 不宜直接信任的领域
  • 复杂依赖安装命令：版本兼容性需人工验证
  • 未经验证的算法结论：模型精度、性能数据需实测
  • 未运行过的完整项目：AI 生成的完整项目首次运行几乎必然报错
  • 隐私安全版权判断：AI 不区分开源协议和版权限制
  • 生产环境配置：安全配置、性能调优需领域知识
核心原则

将 AI 视为"需要代码审查的初级开发者"。其输出必须经过读代码、运行测试、逐行验证三个步骤才能合入项目。

课程要求

每次使用 AI 后提交"AI 使用记录"：任务描述、原始提示词、AI 输出、人工修改点及原因、运行截图。

五类常见问题
  • 幻觉：生成不存在的 API、函数签名或库版本。例如虚构 torchvision.models.yolo

Raw Requirement Analysis

原始需求的特征与局限

原始需求（28 字）
做一个目标检测平台，可以上传图片和视频，调用 YOLO 模型，显示检测结果。

上述描述信息量远远不够。需要拆解出以下维度：

  • 输入：支持的文件格式、单图/批量方式与大小限制
  • 参数：置信度阈值、模型变体等可配置项
  • 输出：检测结果展示形式，包括列表、标注图与 JSON
  • 异常：格式不支持、模型加载失败等场景的处理策略
  • 记录：检测历史的持久化与查询需求
需求拆解方法论

需求拆解遵循"5W1H + 异常覆盖"框架：

  • What：系统目标与核心功能
  • Who：使用者与用户规模
  • Where：运行环境：本地或服务器
  • When：执行方式：同步、异步与排队机制
  • Why：功能价值与优先级
  • How：实现路径与技术选型
拆解原则

每个"正常流程"都对应一个"异常流程"。例如"上传图片"的正常流程对应"格式不支持""文件过大""文件损坏"等异常流程。异常处理本身就是功能需求。

Function Breakdown

功能拆解清单

核心功能
  • 文件上传：支持 jpg / png / mp4 格式，单文件上传，前端预览
  • 参数选择：模型类型（YOLOv5s/v5lite）、置信度阈值（0.0-1.0）、输入尺寸
  • 检测调用：后端加载模型并返回结构化结果（类别、置信度、边界框）
  • 结果展示：原图显示、标注结果图叠加、检测列表（类别/置信度/位置）
  • 视频检测：逐帧分析、关键帧提取、检测结果汇总
辅助功能
  • 记录管理：检测时间、文件名、类别数量、检测结果持久化
  • 结果下载：检测结果图片保存与导出（标注图 + JSON 数据）
  • 批量处理：多文件排队检测，进度显示
  • 历史查询：按日期/文件名/类别筛选检测记录
  • 异常处理：格式错误（400）、模型失败（500）、无目标（空数组）
功能清单的 AI 用途

将功能清单直接粘贴到提示词中作为"背景"部分，AI 能更准确地理解项目范围和边界，减少无关代码生成。

清单粒度建议

每条功能应独立可测试。例如"文件上传"应拆分为"格式校验""大小校验""内容校验"三个独立条目。

规格模板结构
图片检测功能规格md
# 功能：图片目标检测
## 输入
- image: jpg/png，≤ 16MB
- conf: 0.0–1.0，默认 0.25
## 输出
- result_image_url：标注图路径
- detections：[{class, confidence, bbox}]
- elapsed_ms：推理耗时
## 异常
- 文件为空 → 400
- 格式不支持 → 400
- 推理失败 → 500 + 日志
## 验收
- 合法图片 → 200 + detections
- txt 文件 → 400
- 无目标 → 空数组
规格的作用

规格模板起到三重作用：

  • 对齐：团队成员对功能有一致的理解
  • 约束：将规格嵌入提示词，限制 AI 的生成空间
  • 验收：每条验收标准都是可执行的测试用例
规格粒度

课程项目中，每个 API 端点对应一份规格。规格不宜过粗（一个功能一句话）也不宜过细（每行代码都写规格）。输入/输出/异常/验收四部分缺一不可。

Five-Part Structure

提示词的五部分框架

#部分内容作用
1角色"你是一个 Python/FastAPI 视觉应用开发助手"设定回答视角和知识范围
2背景"课程项目使用 YOLOv5-lite 检测图片"提供项目上下文与约束
3输入"已有 detect_image(image_path, conf) 函数"说明现有基础与依赖
4约束"不要改模型代码；返回 JSON"限定 AI 的行为边界
5验收"提供启动命令、测试 curl、目录结构"明确交付物形式和标准
各部分详解

角色决定了 AI 调用的知识子集。指定"FastAPI 开发助手"比不指定角色获得的代码质量更高。背景提供项目上下文，避免 AI 从零假设。输入说明已有代码的接口签名，防止 AI 生成重复或冲突的函数。

约束与验收的关键性

约束是最容易被忽略的部分。缺约束意味着 AI 会在不相关方向上浪费 Token——例如生成了前端代码但只需要后端接口。验收定义了"完成"的标准，避免 AI 输出不完整的半成品。

迭代策略

建议采用"基线提示词 → 评估输出 → 补充约束 → 重新生成"的迭代循环，每次只改一个维度。

常见误区

一次性添加过多约束导致 AI 顾此失彼。应逐步增加约束，观察每次变化对输出的影响。

FastAPI 图片检测接口text
请帮我用 FastAPI 写一个图片检测接口。

【角色】你是 Python FastAPI 视觉应用
开发助手。

【背景】课程项目使用 YOLOv5-lite 检测
图片。前端需要接收 JSON 响应。

【输入】已有 detect_image(path, conf=0.25)
函数，返回 (result_path, detections)。
detections 格式：[{class, confidence, bbox}]

【约束】不要改模型代码；只写后端接口；
返回 JSON；添加异常处理；使用 UUID 命名

【验收】提供 uvicorn 启动命令、curl 测试
示例、最终目录结构。
提示词设计要点
  • 角色具体化：指定"FastAPI 视觉应用开发助手"而非泛泛的"编程助手"
  • 背景包含技术栈：YOLOv5-lite + FastAPI + JSON 响应
  • 输入给出函数签名：明确已有函数的参数和返回值类型
  • 约束可验证："不要改模型代码""UUID 命名""16MB 限制"都是可检查的
  • 验收可执行：curl 命令可以直接运行验证
提示词长度

课程项目中的提示词建议控制在 200-500 字。过短则信息不足，过长则 AI 可能遗漏关键约束。

AI 返回的后端接口代码

app/main.pypython
from fastapi import FastAPI, UploadFile, File, Form, HTTPException
import os, uuid, uvicorn
from pathlib import Path
from detect import detect_image

app = FastAPI(title="目标检测 API")
BASE_DIR = Path(__file__).parent
UPLOAD_DIR = BASE_DIR / "uploads"
RESULTS_DIR = BASE_DIR / "results"
os.makedirs(UPLOAD_DIR, exist_ok=True)
os.makedirs(RESULTS_DIR, exist_ok=True)
MAX_SIZE = 16 * 1024 * 1024  # 16MB

@app.post("/detect/image")
async def detect_image_endpoint(
    file: UploadFile = File(...),
    conf: float = Form(0.25)
):
    # 格式校验
    ext = file.filename.split(".")[-1].lower()
    if ext not in ("jpg", "jpeg", "png"):
        raise HTTPException(400, f"不支持的格式: {ext}")
    # 大小校验
    content = await file.read()
    if len(content) == 0:
        raise HTTPException(400, "文件为空")
    if len(content) > MAX_SIZE:
        raise HTTPException(413, "文件超过 16MB 限制")
    # 安全存储（UUID 防覆盖）
    file_id = str(uuid.uuid4())
    save_path = UPLOAD_DIR / f"{file_id}.{ext}"
    with open(save_path, "wb") as f:
        f.write(content)
    # 模型推理
    try:
        result_path, detections = detect_image(str(save_path), conf)
    except Exception as e:
        raise HTTPException(500, f"推理失败: {str(e)}")
    return {
        "file_id": file_id,
        "result_image_url": f"/results/{Path(result_path).name}",
        "detections": detections,
        "total": len(detections)
    }
约束对应检查

✓ UUID 命名 · ✓ 16MB 限制 · ✓ 异常处理 · ✓ 相对路径 · ✓ JSON 返回 · ✓ 格式校验

仍需人工确认

检查 detect_image 函数的路径参数类型，并确认 results 目录是否通过静态文件服务暴露。

Hallucination · 模型幻觉

风险一：模型幻觉——虚构 API 与函数

虚构 vs 真实代码对比
AI 可能虚构的导入python
# ❌ AI 虚构：此模块不存在
from torchvision.models.yolo import \
    load_pretrained_yolov5

# ✅ 正确的导入方式
import torch
model = torch.hub.load(
    'ultralytics/yolov5',
    'yolov5s', pretrained=True)

AI 还可能虚构方法名，如 model.detect_objects()，而实际 API 是 results = model(img) 后通过 results.pandas().xyxy[0] 获取结果。

幻觉检测方法
  • 文档验证：对每个不熟悉的 API 调用，查阅官方文档确认签名
  • REPL 测试：在 Python 交互环境中逐条执行 import 语句
  • 版本核对：确认 AI 使用的 API 版本与安装的包版本一致
  • 错误信息分析：ModuleNotFoundError 和 AttributeError 是幻觉的直接证据
典型幻觉模式

AI 倾向于将不同库的功能合并到一个虚构的 API 中。例如将 torchvision 的数据集加载功能和 ultralytics 的检测功能合并为一个不存在的 torchvision.detection.YOLO。

Path Errors · 路径与文件结构错误

风险二：路径与文件结构错误

四类路径问题
  • 硬编码绝对路径：/home/user/project/weights/yolov5.pt 在其他机器上必然不存在
  • 错误的工作目录假设：假定脚本在项目根目录运行，但实际可能在 app/ 子目录中
  • 跨平台路径分隔符：Windows 使用 \，Linux/Mac 使用 /
  • 遗漏目录创建：代码写入 uploads/ 但未创建目录，导致 FileNotFoundError
正确做法：pathlib
python
from pathlib import Path

# 基于当前文件定位项目根目录
BASE_DIR = Path(__file__).resolve().parent

# 子目录路径
UPLOAD_DIR = BASE_DIR / "uploads"
RESULTS_DIR = BASE_DIR / "results"
WEIGHTS_DIR = BASE_DIR / "weights"

# 自动创建目录
UPLOAD_DIR.mkdir(exist_ok=True)
RESULTS_DIR.mkdir(exist_ok=True)

# 安全地拼接路径
model_path = WEIGHTS_DIR / "yolov5s.pt"
save_path = UPLOAD_DIR / f"{uuid.uuid4()}.jpg"

使用 pathlib.Path 替代 os.path，自动处理跨平台路径分隔符。

审查要点

搜索代码中的 /home/、C:\Users\、/Users/ 等绝对路径前缀，以及未调用 mkdir(exist_ok=True) 的文件写入操作。

AI 提示词建议

在提示词约束中明确要求"使用 pathlib 处理所有路径""所有路径基于项目根目录""自动创建所需目录"。

AI 生成的 requirements.txt 通常未指定版本号或使用过时版本。建议使用 pip freeze > requirements.txt 生成实际安装的版本。

典型冲突场景
  • PyTorch 与 CUDA 版本不匹配：安装的 PyTorch 版本与系统 CUDA 驱动不兼容
  • opencv-python 与 opencv-contrib-python 冲突：两者不能同时安装
  • Pillow 版本过低：导致 JPEG 编码错误或 WebP 格式不支持
  • numpy 2.x 兼容性：部分旧库不支持 numpy 2.0 的 API 变更
虚拟环境最佳实践

每个项目使用独立的虚拟环境。使用 conda create -n yolo-project python=3.10 创建，conda activate yolo-project 激活。避免在 base 环境中安装项目依赖。

Security Vulnerabilities

风险四：安全漏洞

文件上传安全风险
  • 路径遍历：../../../etc/passwd 作为文件名可能覆盖系统文件
  • 无文件类型校验：仅检查扩展名，不检查 MIME type 和文件魔数
  • 无大小限制：超大文件上传导致磁盘占满或 OOM
  • 覆盖已有文件：使用原文件名保存而非 UUID 重命名
  • 无速率限制：频繁请求可能导致拒绝服务
安全加固代码
python
import magic
import uuid
from pathlib import Path

ALLOWED_MIME = [
    "image/jpeg",
    "image/png",
    "video/mp4"
]
MAX_SIZE = 16 * 1024 * 1024  # 16MB

# MIME type 校验（基于文件内容）
file_type = magic.from_buffer(
    content[:1024], mime=True
)
if file_type not in ALLOWED_MIME:
    raise HTTPException(400, "不支持的文件类型")

# UUID 重命名防覆盖
safe_name = f"{uuid.uuid4()}.{ext}"
save_path = UPLOAD_DIR / safe_name

# 路径遍历防护
if ".." in file.filename:
    raise HTTPException(400, "非法的文件名")
安全审查清单

□ 文件类型校验（MIME + 魔数） □ 文件大小上限 □ UUID 重命名 □ 路径遍历防护 □ 速率限制

依赖安装

MIME 校验需要 pip install python-magic-bin（Windows）或 pip install python-magic（Linux/Mac）。
